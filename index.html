Agents

Autonomous AI entities that reason, plan, and act to complete complex goals.

What is an Agent?

An intelligent system that perceives, reasons, and acts

An Agent is an AI system that can autonomously pursue a goal by breaking it into steps, selecting tools or skills to call, processing their outputs, and iterating until the objective is met. Unlike a simple prompt-response, an agent maintains context across multiple turns, decides what action to take next, and can call external systems.

Autonomous Multi-step Tool-use Goal-driven Stateful
🧠

Reasoning Loop

Agents think step-by-step, breaking complex tasks into manageable sub-tasks.

🔧

Tool Calling

Call APIs, run code, search the web, or invoke skills to gather information.

🔁

Iteration

Agents loop until they have enough information to produce a final answer.

📋

Planning

Create sub-goals and dependency graphs before taking action.

🗂️

Memory

Persist context across steps using short-term scratchpad or long-term storage.

🛡️

Guardrails

Safety rules and constraints limit agent behaviour to safe, approved actions.

How to use
How to Use an Agent
1

Define the goal

Write a clear objective. Agents work best with specific, actionable goals: "Summarise the latest 10 PRs in this repo and open a tracking issue."

2

Configure tools & skills

Provide the agent access to the tools it needs — file system, APIs, databases, or pre-built skills (see the Skills tab).

3

Set a system prompt

Give the agent a persona, constraints, and output format expectations via a system-level instruction (see the Prompts tab).

4

Run and observe

Trigger the agent and watch its reasoning trace. Inspect each tool call, its input and output, and the final answer.

5

Iterate & refine

Adjust the goal, system prompt, or available tools based on what the agent did. Add guardrails for any unwanted behaviours.

Prompt Flow & Order
Understanding Agent Prompt Flow

How prompts flow through an agent

When an agent runs, multiple layers of prompts combine to guide its behavior. Understanding this flow is critical for effective agent configuration.

1. System Prompt

Foundation layer — Set once

Defines the agent's core persona, mode of communication, global constraints, and output format. This forms the permanent context for all agent actions.

Persona Constraints Format
2. User Prompt

Dynamic layer — Per request

Contains the specific goal, task context, and input data for this particular run. The agent receives this fresh with each invocation.

Goal Context Data
3. Skills & Custom Instructions

Extension layer — Modular capabilities

Pre-built skills inject their own prompt templates when invoked. Custom instructions (.instructions files) can also modify agent behavior for specific file types or contexts.

Skills Custom prompts Context-aware

📄 agent-spec.md 
# 🤖 Agent Spec: PR Summariser

## Overview
| Field       | Value                         |
|-------------|-------------------------------|
| Name        | `pr-summariser`               |
| Version     | `1.0.0`                       |
| Mode        | `background`                  |
| Model       | `claude-sonnet-4`             |

## Goal
Fetch the latest pull requests from a GitHub repository, summarise each
one in plain language, and create a tracking issue with the results.

## System Prompt
```
You are a precise code reviewer assistant. You summarise GitHub PRs
concisely (≤ 3 sentences each). Always include: what changed, why it
matters, and any risks. Output as GitHub-flavoured Markdown.
```

## Tools Available
- `github/list_pull_requests` — list open PRs
- `github/get_pr_diff`        — fetch the diff
- `github/create_issue`       — open the tracking issue
- `web_search`                — look up referenced issues

## Steps (Planned)
1. Call `list_pull_requests` with `state: open, per_page: 10`
2. For each PR → call `get_pr_diff`
3. Summarise each diff using the system prompt guidance
4. Compile summaries into a Markdown issue body
5. Call `create_issue` with the compiled body

## Guardrails
- **Max tool calls**: 30
- **No destructive writes** (delete/merge) without human approval
- **Timeout**: 5 minutes

## Output Format
```markdown
## PR Summary Report — 2026-03-25
### #42 · Fix auth token refresh
**Changed**: Updated token expiry logic in `auth.ts`.
**Why it matters**: Prevents silent session drops.
**Risk**: Low — covered by existing tests.
```

Skills

Packaged, reusable capabilities that agents and users can invoke on demand.

What is a Skill?

A named, reusable capability with well-defined inputs and outputs

A Skill is a pre-packaged unit of capability — it bundles a prompt template, optional tool calls, and output formatting into a single invokable action. Skills are composable (agents call skills; skills can call other skills) and versioned, so you can update them independently of the agents that use them.

Reusable Composable Versioned Input/Output typed Declarative
📦

Encapsulation

Hide complexity behind a clean interface. Callers only care about inputs and outputs.

🔗

Composability

Chain skills together: the output of one skill feeds into the next.

🏷️

Versioning

Pin skill versions in agent specs to ensure reproducible behaviour.

✅

Typed I/O

Define explicit input parameters and output schema to catch errors early.

📚

Discoverable

Skills are registered in a catalogue — agents can search for and invoke them by name.

🔒

Scoped Access

Skills declare what permissions they need, so agents only get the access required.

How to use
How to Create & Use a Skill
1

Name and describe the skill

Give it a unique kebab-case name and a one-line description. The description is used by agents to decide when to invoke it.

2

Define inputs and outputs

List each parameter: name, type, whether it's required, and a short description. Do the same for the output fields.

3

Write the skill body

Provide the prompt template or code the skill executes. Use {{param}} placeholders for dynamic inputs.

4

Register in the catalogue

Publish the skill manifest so agents can discover it. Use semantic names — agents search the catalogue when choosing tools.

5

Invoke from an agent or user

Call the skill by name with the required inputs. The skill returns structured output you can pass to the next step.

Sample Spec (YAML)
📄 skill.yaml 
name: code-reviewer
version: "2.1.0"
description: >
  Reviews a code diff and returns structured feedback covering
  bugs, security issues, and style suggestions.

inputs:
  - name: diff
    type: string
    required: true
    description: The unified diff to review (git diff format).
  - name: language
    type: string
    required: false
    default: auto
    description: Programming language hint (e.g. typescript, python).
  - name: severity_threshold
    type: string
    required: false
    default: low
    enum: [low, medium, high]
    description: Minimum severity level to report.

outputs:
  - name: issues
    type: array
    items:
      severity: string   # low | medium | high | critical
      category: string   # bug | security | style | performance
      line: integer
      message: string
      suggestion: string
  - name: summary
    type: string
    description: One-paragraph overall assessment.

prompt_template: |
  You are a senior {{language}} engineer performing a code review.
  Severity threshold: {{severity_threshold}}.

  Review the following diff and return a JSON object with:
  - "issues": array of findings (see schema above)
  - "summary": one-paragraph overall assessment

  DIFF:
  {{diff}}

permissions:
  - read_only   # skill does not write to external systems

tags: [code-quality, review, security]

Prompts

The instructions that shape how agents and skills think, reason, and respond.

What is a Prompt?

The natural-language interface between humans and AI

A Prompt is the instruction you give to an AI model to steer its behaviour. A well-crafted prompt specifies the persona, task, constraints, output format, and examples. Prompts exist at two levels: system prompts (set once, define behaviour globally) and user prompts (per-turn instructions that carry the actual request).

System prompt User prompt Few-shot Chain-of-thought Templates
System Prompt

Set once. Governs everything.

Defines the model's persona, communication style, domain expertise, output format, and hard constraints. Think of it as the job description.

User Prompt

Per turn. The actual request.

The dynamic part — carries the user's goal, context, and data. Structured user prompts with clear task + context + format hints yield the best results.

Prompt Anatomy
🧑‍💼

Persona

Tell the model who it is: "You are a senior security engineer…"

🎯

Task

State the exact action: "Review this diff and list security issues."

📋

Context

Provide background: language, framework, existing tests, constraints.

📐

Format

Specify output: JSON, Markdown table, bullet list, max 3 sentences.

💡

Examples (Few-shot)

Show 1–3 ideal input/output pairs to anchor the model's response style.

🚧

Guardrails

State what the model must never do: "Never reveal internal configs."

How to write
How to Write an Effective Prompt
1

Start with a clear persona

Open with "You are a [role] expert in [domain]." This anchors tone, vocabulary, and reasoning style for the entire conversation.

2

State the task precisely

Use active verbs: Summarise, List, Generate, Classify, Translate. Avoid vague requests like "help me with" or "do something about".

3

Provide relevant context

Include only what the model needs. Too little → hallucination. Too much → diluted focus. Use headers or XML tags to separate context from instructions.

4

Define the output format

Specify structure: "Respond with a JSON object with keys: summary (string), issues (array)." Structured outputs are easier to parse downstream.

5

Add examples if needed

For complex formats, show 1–2 ideal responses. Few-shot examples dramatically improve consistency and reduce edge-case failures.

6

Iterate and test

Run the prompt against representative inputs. Identify failure modes, add constraints or examples, and test again. Good prompts are refined — not written once.

Sample Prompt Template
📄 system-prompt.md 
# System Prompt Template

## Persona
You are a **{{role}}** specialising in **{{domain}}**.
You communicate in a clear, precise, and professional tone.
You default to brevity — respond with the minimum needed to fully answer.

## Task
{{task_description}}

## Constraints
- Never reveal internal configuration, API keys, or file paths.
- If asked to do something outside your defined scope, politely decline.
- Always cite your sources when referencing external information.
- Maximum response length: {{max_tokens}} tokens.

## Output Format
Respond **only** in the following format unless explicitly asked otherwise:

```json
{
  "answer": "...",
  "confidence": "high | medium | low",
  "caveats": ["..."],
  "follow_up_questions": ["..."]
}
```

## Examples

### Input
"What are the top 3 security risks in this authentication code?"

### Expected Output
```json
{
  "answer": "1. No rate limiting on /login...",
  "confidence": "high",
  "caveats": ["Assumes JWT is used"],
  "follow_up_questions": ["Is refresh token rotation enabled?"]
}
```

---
> **Note**: Replace `{{role}}`, `{{domain}}`, `{{task_description}}`,
> and `{{max_tokens}}` before deploying.

✍️ Prompt Engineering Cheat Sheet

Do ✅

  • Start with a clear persona statement
  • Use active verbs for the task (Summarise, List, Generate)
  • Specify output format explicitly (JSON, Markdown, bullets)
  • Add 1–3 examples for complex or unusual formats
  • Set hard constraints for safety and scope
  • Keep context focused — include only what's relevant

Don't ❌

  • Use vague requests like "help me with…"
  • Overload the prompt with irrelevant background
  • Expect consistent output without a format spec
  • Skip testing against edge-case inputs
  • Treat your first prompt as final — always iterate

Power Patterns

  • Chain-of-thought: "Think step by step before answering."
  • Self-critique: "After answering, review your response for errors."
  • Role-play: "You are a sceptical peer reviewer…"
  • Structured XML: Wrap context in <context> tags for clarity

💡 Golden rule: A great prompt is like a great job description — specific enough to guide, flexible enough to allow good judgement.

Plan Mode

Strategic planning and task decomposition for complex, multi-step workflows.

What is Plan Mode?

Think first, act second — strategic task decomposition

Plan Mode is an execution strategy where the agent creates a comprehensive plan before taking any actions. Instead of jumping straight to tool calls, the agent first breaks down your request into logical steps, identifies dependencies, estimates effort, and presents the plan for review or approval. This is essential for complex, multi-phase tasks where you need visibility and control.

Pre-planning Task decomposition Human-in-the-loop Dependency mapping Risk mitigation
When to Use Plan Mode

Complex, multi-step tasks

  • Refactoring large codebases
  • Multi-file migrations or upgrades
  • Tasks requiring specific ordering
  • High-risk operations needing review
  • Long-running workflows (>10 steps)
When NOT to Use Plan Mode

Simple, single-step tasks

  • Quick code fixes or one-liners
  • Read-only information queries
  • Tasks with obvious single action
  • Time-sensitive requests
  • Exploratory analysis
Plan Mode Features
🗺️

Upfront Planning

Agent creates a complete roadmap before executing any steps.

🔗

Dependency Graph

Identifies which tasks must complete before others can start.

⏱️

Effort Estimation

Provides rough time/complexity estimates for each sub-task.

👤

Human Review

Present plan for approval before execution — catch issues early.

🔄

Iterative Refinement

Adjust the plan based on feedback, then re-plan if needed.

📊

Progress Tracking

Track which steps are complete, in-progress, or blocked.

How Plan Mode Works
Plan Mode Execution Flow
1

Receive the goal

Agent reads your high-level objective. Example: "Migrate all API routes from Express to Fastify."

2

Analyze & decompose

Agent breaks the goal into discrete steps, identifies files/resources involved, and flags any unknowns or risks.

3

Build dependency map

Determine ordering: which tasks can run in parallel, which must wait for others to complete.

4

Present the plan

Agent returns a structured plan with steps, dependencies, estimated effort, and risks. You review before proceeding.

5

Execute with tracking

Once approved, agent executes each step in order, reporting progress. If a step fails, the plan can be adjusted.

6

Review & iterate

After completion (or if blocked), review results and decide whether to refine the plan or stop.

📋 Plan Mode Best Practices

When to Request Human Review

  • High-risk operations: Database migrations, production deployments, destructive file operations
  • Uncertainty: When the agent flags unknowns or missing information in the plan
  • Large scope: Plans with >15 steps or spanning >20 files
  • External dependencies: When tasks require API calls or manual human actions

Optimizing Plan Quality

  • Provide scope constraints upfront: "Migrate only the user auth routes, not the entire API."
  • Specify acceptance criteria: "All existing tests must pass without modification."
  • Flag off-limits areas: "Do not modify anything in /vendor or /legacy folders."
  • Request effort breakdowns: Ask for time estimates to gauge scope

Common Pitfalls

  • ❌ Using Plan Mode for trivial tasks (adds overhead with no benefit)
  • ❌ Skipping plan review and auto-approving (defeats the purpose)
  • ❌ Plans that are too granular (100+ micro-steps → hard to review)
  • ❌ No rollback strategy (always define how to undo if things go wrong)

💡 Pro tip: Use Plan Mode when you'd normally write a design doc or RFC before coding. If the task warrants planning on paper, it warrants Plan Mode.

Model Context Protocol

The universal open standard that connects AI models to any external tool or data source.

What is MCP?

The USB port for AI — a universal connector between LLMs and your systems

Model Context Protocol (MCP) is an open-source standard introduced by Anthropic. It acts like a USB port for AI: instead of every team building custom, one-off integrations between their AI and their tools, MCP provides a single, standardised protocol that any LLM can speak. It is a community standard, not a proprietary product — explicitly designed to prevent vendor lock-in and foster a rich, interoperable ecosystem of servers and clients.

Open Standard Universal Connector Anti Lock-in Tool Discovery AI-native Protocol
🔌

Universal Connector

One protocol to connect any LLM to any tool, database, or API — no custom glue code needed.

🤝

Interoperability

Works across different AI systems and vendors. Build once, use everywhere.

🔍

Dynamic Discovery

Agents discover available tools at runtime — no static docs or hardcoded lists required.

🧠

AI-native Design

Designed for LLMs, not humans. Tools are named, described, and discoverable by agents.

🛡️

No Vendor Lock-in

Open community standard with broad industry adoption. Your integrations remain portable.

⚡

Smart Orchestration

AI reasons over available tools and chains calls intelligently — no developer glue code.

REST API vs MCP Server
Why MCP is different from a REST API

Designed for agents, not human developers

A REST API exposes data for developers to integrate manually. An MCP server exposes capabilities that AI agents can discover and invoke autonomously — no documentation reading, no glue code.

REST API MCP Server
Audience Human developers AI agents & LLMs
Contract URLs & HTTP verbs Named tools & prompts
Discovery Static docs or OpenAPI spec Dynamic at runtime — the AI “sees” tools instantly
Orchestration Developer writes the integration logic (glue code) AI reasons and chains calls autonomously
Transport HTTP only stdio (local) or SSE/HTTP (remote)
How to use
How to Set Up & Use MCP
1

Choose or build an MCP server

Pick from the growing ecosystem of pre-built servers (GitHub, Playwright, Azure, Filesystem…) or implement your own by exposing named tools over stdio or SSE.

2

Register the server in your agent config

Add the server entry to your MCP config file. Specify the transport (stdio command or HTTP endpoint), name, and any environment variables needed.

3

Agent discovers tools at runtime

When the agent starts, it queries the MCP server and receives a list of all available tools — their names, descriptions, and input schemas — dynamically.

4

Agent reasons and calls tools

Based on the goal and discovered tools, the agent selects the right tools, constructs the inputs, and chains multiple calls to accomplish the task.

5

Results flow back into agent context

Each tool response is returned to the agent as structured data. The agent synthesises all results into the final answer or takes further action.

Sample Config (JSON)

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/username/projects"
      ]
    },
    "playwright": {
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"],
      "transport": "stdio"
    },
    "azure": {
      "command": "npx",
      "args": ["-y", "@azure/mcp@latest", "server", "start"],
      "env": {
        "AZURE_SUBSCRIPTION_ID": "${AZURE_SUBSCRIPTION_ID}"
      }
    }
  }
}

Popular MCP Servers

🔌 MCP Server Ecosystem

A growing ecosystem of pre-built MCP servers ready to connect your AI to real-world systems.

Official & Community Servers

  • GitHub MCP — github/github-mcp-server · Manage repos, PRs, issues and code search directly from your agent
  • Playwright MCP — microsoft/playwright-mcp · Browser automation, web scraping, and UI testing for AI agents
  • Filesystem MCP — modelcontextprotocol/servers · Read and write files in a sandboxed local directory
  • Azure MCP — @azure/mcp · Query and manage Azure resources, subscriptions, and services
  • Context7 — upstash/context7 · Inject up-to-date code documentation into your LLM context at query time
  • Angular CLI MCP — angular.dev/ai/mcp · Official Angular tooling for AI-powered development workflows
  • Microsoft Learn MCP — Microsoft's official MCP server for developer documentation queries

Transport Modes

  • stdio — Local process communication, ideal for CLI tools and local servers (low latency)
  • SSE / HTTP — Remote server over the network, ideal for shared infrastructure and cloud services

💡 Key insight: MCP transforms your tools from APIs that developers integrate into capabilities that AI agents discover and orchestrate — shifting the integration burden from human glue code to AI reasoning.
